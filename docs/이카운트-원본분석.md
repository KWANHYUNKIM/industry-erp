1. 도메인 / 세션 / 전체 아키텍처 / 데이터 연결 방식
[1-1] 도메인 구성
* 애플리케이션 도메인: loginad.ecount.com (로그인 및 ERP 실행, 모든 데이터 API 호출이 이 도메인에서 이루어짐)
* - 정적 리소스 CDN: resourcev3.ecount.com (JS 번들, 위젯, 언어 리소스 등 화면 구성 파일 제공)
* - ERP 실행 진입점 URL: https://loginad.ecount.com/ec5/view/erp
[1-2] 세션 / 식별자 체계
* ec_req_sid: 현재 세션 요청 식별자 (예: AD-ETdns3ELqTxfQ). 모든 데이터 요청 URL에 부착됨.
* - tenant_sid: 회사(테넌트) 식별자 = 679546 (주식회사 팜인)
* - user_sid: 로그인 사용자 식별자 (예: KWANHYUN950326)
* - 이 세 값의 조합으로 ‘어느 회사의, 어느 사용자가, 어느 세션에서’ 요청했는지 구분함.
[1-3] 화면 라우팅 구조 (SPA)
* 전체가 단일 페이지 앱(SPA)이며, URL 해시(#)로 화면을 전환함.
* - 해시 파라미터: menuType, menuSeq, groupSeq, prgId(프로그램ID), depth(메뉴 깊이)
* - 예: 판매입력 = prgId=E040205, 재고현황 계열 = ESZ, 거래처등록 = prgId=E010101
* - 대분류/중분류는 MENUTREE_000004(재고 I) 처럼 MENUTREE_xxxxxx 코드로 식별됨.
[1-4] 데이터 연결 방식 (API)
* 좌측 메뉴 로딩: /ec5/api.bulk/app.setup/action/menu_setting/GetNavigationLeftMenuAction (GET) — sid, first_depth_sid로 해당 메뉴 트리 반환
* - 실데이터 조회/저장: /ECERP/SVC/{모듈}/{프로그램코드} (POST) — 예: /ECERP/SVC/ESD/ESD006M(판매입력), /ECERP/SVC/ESZ/ESZ001R(재고현황)
* - 화면 양식(폼/그리드 정의) 캐시: /ECERP/SVC/Popup/GetXFormCache, GetXFormCacheSearch, GetXFormCacheCalcNew, GetXFormCacheHashKeyV2
*   → ‘화면 양식 정의’와 ‘실제 데이터’가 분리되어 관리됨 (양식은 캐시, 데이터는 실시간 조회)
* - 빠른입력 캐시: /ECAPI/SVC/Common/Infra/GetListQuickEntryCache
[1-5] UI 프레임워크
* 자체 위젯 프레임워크 ‘eccomposite_v1’ 사용. 화면은 header(조건영역) + contentsform/grid(명세·결과) + footer(액션버튼)의 3단 구조로 통일됨.
* - 모듈별 번들을 필요 시 로드: core, inventory, account, approval, basic, output 등
* - 언어 리소스는 ko-kr.* 파일로 분리 제공 (다국어 지원 구조)
[1-6] 핵심 원리
* 모든 기능은 ‘전표 입력 → 수불/전표 생성 → 현황·장부에 집계’로 흐름. 기준정보(거래처·품목)는 재고·회계가 공유하는 단일 마스터.

2. 재고 I 모듈

[2-2] 기초등록

* - 품목등록(prgId=E040103): 리스트 컴럼 = 품목코드·품목명·이미지·구매처명·품목구분([상품]/[원재료]/[부재료])·규격정보·재고수량관리(수량관리대상/제외)·품목그룹1명·검색장내용·사용(YES/NO)·파일관리
*   · 하단 버튼: 신규(F2)·바코드·관계설정·계층그룹·변경·재고조정·사용중단/재사용·Excel / 페이징(14페이지)
*   · 상태값: ‘수량관리대상’ 품목만 재고 수불에 반영, ‘수량관리제외’은 금액만 관리(수량 미관리)
[2-3] 영업관리
* 흐름: 견적서 → 주문서 → 판매 → 수금 (+ 출하지시서·출하)
* - 판매입력(prgId=E040205, 서비스 ESD002. 재고 I6M): 헤더 = 일자·거래처·담당자·출하창고·거래유형(기본 ‘부가세율적용’)·통화(기본 ‘내자’)·프로젝트
*   · 명세 그리드: 품목코드·품목명·규격·시리얼/로트·수량·단가·공급가액·부가세·부대비용·적요 (하단 합계행 자동집계)2. 재
*   · 도구 버튼: 찾기(F3)·소요·주문·구매(다른 전표 불러오기)·보류·할인·재고불러오기·바코드·검증·이익계산·전표불러오기·거래별부가세계산
*   · 저장 버튼: 저장(F8)=재고수불만 발생 / 저장·전표(F7)=재고수불+회계전표 동시 생성 / 회계전표연결·현금수금
*   · 상태값 핵심: 판매 입력 시 창고에서 출고 발생 → 재고 차감
[2-4] 구매관리
* 흐름: 발주요청 → 발주계획 → 단가요청 → 발주서 → 구매 → 지급
* - 구매입력(prgId=E040303): 판매입력과 동일 구조, 단 ‘출하창고’ 대신 ‘입고창고’, 그리드 컴럼 = 품목코드·품목명·규격·기본수량·단가·공급가액·부가세·적요, 도구 버튼에 발주·주문 불러오기
*   · 상태값 핵심: 구매 입력 시 창고에 입고 발생 → 재고 증가
[2-5] 생산/외주
* 하위: BOM(소요량)·공정·생산계획/MRP생성·작업지시서·생산불출·작업·생산입고(조회·I/II/III·현황·소모현황)·지급현황·외주비확인현황·외주비회계반영
* - 흐름: 작업지시서 → 생산불출(소모자재를 공장으로 이동) → 생산입고(제품/반제품 입고 + 소모품목 차감)
* - 생산입고조회: 상태 필터 탭(전체·결재중·미확인·확인), 컴럼 = 일자-No.·생산완공장명·받는창고명·품목명[규격]·수량·담당자명·작업지시서·인쇄
*   · 상태값: 결재중 / 미확인 / 확인 (생산 진행 상태)
[2-6] 기타이동 / 출력물
* 기타이동: 창고간 이동·자가사용·불량처리·대체사용·폐기처리 등 재고 증감 사유를 기록
* - 출력물(조회 전용): 재고현황·창고별·관계별·다규격별재고현황, BOM환산재고, 재고잔량분석표, 재고수불부, 재고변동표, 일보 등
*   · 재고현황(ESZ001R): 조건=기준일자·창고·품목·옵션(수량관리제외포함/사용중단포함/안전재고미만표시), 결과=품목코드·품목명[규격]·재고수량 (마이너스 재고는 분홍색 강조)
*   · 재고수불부(E040702): 기간·창고·품목 + 단가표시 옵션(판매/구매/기타단가 각각 단가·vat포함·월별원가·출고/입고단가 선택) → 수불 금액 계산 방식 결정
*   · 결과값 산출: 재고수량 = 기초잔액 + 구매입고 + 생산입고 − 판매출고 − 생산소모 ± 재고조정 ± 창고이동

■ A-4. 재고 결과값 산출 로직(핵심 공식) 재확인
재고현황의 ‘재고수량’은 특정 시점까지의 모든 수불 트랜잭션 누적으로 계산된다:
  재고수량 = 기초잔액 + (구매입고 + 생산입고 + 기타입고 + 창고이동입고)
            − (판매출고 + 생산소모 + 기타출고 + 창고이동출고) ± 재고조정
  → 음수 재고는 재고현황(ESZ001R) 그리드에서 분홍색으로 강조 표시(마이너스 재고 경고).
재고금액(평가액)은 원가법 설정(이동평균/총평균/선입선출 등 품목·회사 정책)에 따라 단가가 결정되며, 재고수불부(E040702)에서 입/출고별 단가·금액과 잔액을 시계열로 확인한다. 판매·구매·생산 입력이 곧바로 이 수불 원장에 반영되어 현황·수불부·재고자산 회계금액이 하나의 원천에서 파생되는 단일 데이터 소스(single source of truth) 구조다.

· 흐름1(영업→회계): 판매전표 → (회계전표연결) → 회계전표(GL) → 채권(AR) 증가, (세금)계산서 → 부가세(VAT).
· 흐름2(구매→회계): 구매전표 → 회계전표(GL) → 채무(AP) 증가 → 자금지급 시 AP 감소.
· 흐름3(재고): 판매/구매/생산 → 재고수불(STOCK) → 재고현황·이익관리(원가).
· 흐름4(인사→세무): 근무기록 → 급여(PAYROLL) → 원천징수(WHT) → 지급명세서.
· 흐름5(결재): 지출결의서·전표 → 전자결재(APPROVAL) 승인 → ERP전표 확정.
· 요약: 개념 스키마는 기준정보(마스터) + 전표(헤더·명세 1:N) + 파생 집계(원장·재무제표·채권채무·세무신고)의 3계층으로 구성된다.
