# QA 시드 · 시나리오 검증

핵심 업무 흐름이 실제로 동작하는지 API를 끝까지 몰아보며 확인합니다. 의존성 없이 Node 18+ 만 있으면 됩니다.

## 실행

```bash
# 백엔드(8081)와 Postgres가 떠 있어야 합니다
node qa/qa.mjs          # 시드 + 검증 (기본)
node qa/qa.mjs seed     # QA 마스터 데이터만 생성
```

통과하면 exit 0, 하나라도 실패하면 실패 항목을 출력하고 exit 1로 끝납니다. CI에 그대로 물릴 수 있습니다.

환경변수로 대상을 바꿀 수 있습니다: `ERP_API`(기본 `http://localhost:8081/api`), `ERP_USER`, `ERP_PASS`.

## 무엇을 검증하는가 (단언 294개)

| 시나리오 | 확인하는 것 |
|---|---|
| 1. 수주 → 출하 → 미출고현황 | 부분출하, **출하지시(READY)는 아직 출하가 아님**, 초과출하 거부, 전량출하 시 주문 완료, 출하취소 시 수량·상태 롤백 |
| 2. 생산계획 → 작업지시 | 문자열이 아닌 실제 FK로 연결됨, 같은 계획 재지시 거부 |
| 3. 생산실적 → 재고 | BOM대로 자재 자동 출고 + 완제품 입고 (백플러시) |
| 4. 품질검사·작업내역 | 마스터에 있으면 FK 연결, 없으면 `null` + 입력 문자열 보존 |
| 5. 환경설정·보안정책 | 저장 후 재조회해도 값이 유지됨(실제 영속) |
| 6. 견적 → 발송 → 수주전환 | 부가세 10% 자동계산, 전환 시 수주 생성·FK 연결·금액 승계, 재전환/취소건 전환 거부 |
| 7. 발주 → 단가확정 → 입고전환 | 단가 미입력 시 기준단가 대체, 확정단가로 금액·부가세 재계산, 입고전환 시에만 재고 증가·구매전표 FK 연결, 단계 건너뛰기/재입고/취소건 입고 거부 |
| 8. 기타이동 | 자가사용·불량처리는 차감, 재고조정은 실사수량과의 차이만큼 증감, 재고부족·차이없음 거부 |
| 9. 급여 원천징수 → 이행상황신고서 | 소득세·지방소득세(소득세의 10%) 자동 공제, 확정 전에는 신고 대상 제외, 신고서·연간 영수증 금액이 급여 공제액과 일치 |
| 10. 계좌/카드 | 입출금이 잔액에 반영·잔액 초과 출금 거부, 카드사용 부가세 10% 자동, 차)비용·부가세대급금 / 대)미지급금 분개 |
| 11. 고정자산 | 정액법 월 상각액 = (취득가−잔존가)/내용연수/12, 차)감가상각비 / 대)감가상각누계액, 같은 달 재실행 시 이중상각 없음, 처분손익 = 처분가−장부가 |
| 12. 어음 | 수취/발행 분개(받을어음↔외상매출금, 외상매입금↔지급어음), 만기결제 시 계좌 잔액 증감, 할인료는 매출채권처분손실, 부도는 현금 없이 외상매출금 환원, 지급어음 할인·부도 거부 |
| 13. FastEntry 간편전표 | 지출결의서 차)비용들/대)결제수단, 입금보고서 차)입금수단/대)수입들, 가지급금정산서 반납·추가지급 처리, 계좌로 처리하면 계좌 잔액도 함께 움직임 |
| 14. 비현금거래(대체전표) | 유형이 차/대변을 정함(상계·대손·미지급 계상·계정대체), 현금성 계정·동일 계정은 거부 |
| 16. 공용메일 | 자기 자신 발송 거부, 받는 사람만 읽음 처리, 공용메일은 담당자 배정 후에만 처리 가능하고 배정된 담당자만 처리, 처리 시 미처리 건수 감소, 재처리·재배정 거부 |
| 17. 수표관리 | 수취 차)받을수표/대)외상매출금, 입금 시 계좌 잔액 증가, 부도는 계좌를 건드리지 않고 채권 환원, 발행 시 예금 즉시 차감, 타입에 맞지 않는 처리·중복 수표번호 거부 |
| 18. 계약관리·전자계약 | 작성→서명요청→전자서명→해지 순서 강제(단계 건너뛰기·재서명·재해지 거부), 서명자·서명일시·동의문구 보관, 종료일<시작일 거부 |

| 21. 수출 | 외화 인보이스가 발행일 고시환율로 원화 고정, 단계 건너뛰기·되돌리기·같은 단계 반복 거부, 입금 시 미입금 잔액에서 빠짐, 매입처 수출 거부 |
| 22. 전자결재 설정 | 사용중인 양식 삭제 거부(사용중지는 가능·기안 목록에서만 빠짐), 결재선 순서 저장·수정, 같은 결재자 연속·빈 결재선 거부 |
| 19. 수입비용 | 수익 계정이 아니면 등록 거부, 회수 수단이 분개 차변을 정함(현금/예금/외상매출금), 계좌입금은 잔액 증가·삭제 거부, 계정별 구성비와 순수지 |
| 15. 예산·자금계획 | 같은 달 같은 계정 이중 편성 거부, 집행실적은 회계전표에서 집계(전표를 끊으면 집행률이 따라옴), 집행액이 편성액을 넘으면 초과 표시, 자금 계획 합계 대비 계좌 실적 수지 |

## 반복 실행

마스터 데이터(`QA-` 접두사: 창고·거래처·품목·BOM·공정·로트)는 **있으면 재사용**하므로 몇 번을 돌려도 안전합니다.
다만 전표(수주·출하·생산실적·검사)는 실행할 때마다 새로 쌓입니다. 삭제 API가 없는 전표가 있어 스크립트가 지우지 않습니다.

깨끗한 상태에서 다시 시작하려면 DB를 통째로 초기화하세요.

```bash
docker compose down -v && docker compose up -d
# 백엔드 재기동 → Flyway가 V1~V3를 실행해 스키마를 새로 만들고, 기본 시드가 들어갑니다
cd backend && ./mvnw spring-boot:run
```

> ⚠️ `down -v`는 **DB를 완전히 지웁니다.** 남겨야 할 데이터가 있으면 먼저 백업하세요.
> ```bash
> docker exec erp-postgres pg_dump -U erp erp > backup.sql
> ```

## 스키마를 바꿀 때

`ddl-auto: validate` 입니다. **엔티티만 고치면 기동이 실패합니다.** 반드시 마이그레이션을 함께 추가하세요.

1. `backend/src/main/resources/db/migration/V<다음번호>__설명.sql` 추가
2. 백엔드 재기동 → Flyway가 적용
3. `node qa/qa.mjs` 로 회귀 확인

`V1__baseline_schema.sql`은 수정하지 마세요. 이미 적용된 마이그레이션을 고치면 체크섬이 어긋나 기동이 실패합니다.

## 프론트 타입체크

```bash
cd frontend && npm run typecheck     # tsc --noEmit -p tsconfig.app.json
```

> ⚠️ **`npx tsc --noEmit` 만 치면 아무것도 검사하지 않고 통과합니다.**
> 루트 `tsconfig.json` 은 `"files": []` + project references 구조라, 프로젝트를 지정하지 않으면
> 검사 대상이 0개입니다. 반드시 `npm run typecheck`(또는 `npm run build`)로 확인하세요.
